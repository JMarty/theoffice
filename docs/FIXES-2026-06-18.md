## P0 — reliability core (the real risk; affects the live Argus/gemini path)

| # | Problem | Fix | Where |
|---|---------|-----|-------|
| 1 | A wedged `codex exec` / `agy --print` never fires `close`, so the agent stays `inFlight` forever and the deliverer skips every future message — only an engine restart heals it. | Per-turn node-side watchdog (`SIGKILL` on overrun) on both exec runtimes, with a single `settled` latch so close/error/timeout settle exactly once and `clearTimeout` always runs. `spawn()` wrapped in try/catch so a synchronous throw frees `inFlight` + requeues. | `src/session/codex-runtime.ts`, `src/session/gemini-runtime.ts`, `src/session/exec-outcome.ts` (+test) |
| 2 | `markDelivering` flips an item to `'delivering'` before the async turn; a crash/restart mid-turn (incl. the self-update restart) strands it there forever — never retried, failed, or surfaced. | Boot reaper `requeueStaleDelivering()` runs once at boot, after `openDb`, before the deliverer (single-writer → race-free). Keeps the charged attempt so a poison item still fails at the cap. | `src/queue/index.ts`, `src/index.ts` (+`src/queue/index.test.ts`) |
| 3 | gemini took ANY exit-0 as success, so a usage cap or an empty turn was silently marked "delivered" and the message lost. | Success now requires exit 0 **and** real STDOUT output; usage caps are held (`requeueNoPenalty`). **stderr only feeds the usage signal, never the output signal** (review fix — a deprecation/progress line on stderr must not look like an answer). | `src/session/gemini-runtime.ts`, `src/session/exec-outcome.ts` |
| 4 | The 2s deliverer tick fires even while the previous async (claude) delivery runs, so an overlapping tick could inject the same prompt twice. | Reentrancy guard (`running` flag in try/finally) serializes ticks. | `src/session/session-manager.ts` |
